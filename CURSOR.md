# 커서 작업 지시서 (freepasserp4)

> **작업 시작 전 이 문서를 먼저 읽는다.**
> 함께 읽을 것: **`DESIGN.md`(디자인 일관성 규격 — 색·타입 스케일·컨트롤 토큰)**, `CLAUDE.md`(규격 SSOT 최종 기준).
> 이 문서는 "커서가 반드시 지킬 것"만 추린 것이다.

---

## 0. 한 줄 원칙

**커서는 "배열"을 한다. "디자인"과 "설계"는 하지 않는다.**
막히면 스스로 새로 만들지 말고 **멈추고 보고**한다. 새로 만드는 순간 누더기가 된다.

---

## 1. 레인 (무엇을 하고, 무엇을 안 하는가)

### ✅ 해도 되는 것
- 기존 공용 원자·엔진을 **가져다 배치**하기 (페이지 조립, 패널 배열, 배선)
- 반복 치환, 이름 변경, import 정리, 타입 오류 수정
- 이미 정해진 규격을 **그대로 적용**하기
- 데이터 배선(어떤 값 → 어떤 원자에 넣기)

### ❌ 하면 안 되는 것
- **디자인 판단** (간격·타이포·색·행 높이·아이콘 도입·카드 구성 변경)
- **공용 원자를 새 컴포넌트로 대체**하기
- 없는 원자를 **손으로 만들어 쓰기**
- 아키텍처·데이터 모델·상태 흐름 변경
- "더 좋아 보여서" 하는 임의 개선

## 2. 절대 금지 (빨간 줄)

### 🚫 2-1. UI를 손으로 그리지 않는다
`<input>` `<textarea>` `<select>` `<button>`을 **직접 쓰지 않는다.** 반드시 공용 원자를 쓴다.

| 필요한 것 | 써야 할 원자 |
|---|---|
| 한 줄 입력 | `Input` |
| 여러 줄 입력 | `Textarea` |
| 검색창 | `SearchInput` |
| 버튼 | `Btn` / `IconBtn` |
| 선택 | `Select` |
| 상태 표시 | `Badge` |
| 목록 행 | `ListRow` (또는 해당 목록의 지정 행) |
| 패널 | `PaneHead` / `PaneBody` / `Panel` |

**예외:** 숨김 파일 선택(`<input type="file" style={{display:'none'}}`)만 허용.

> **필요한 원자가 없으면?** → 손으로 만들지 말고 **멈추고 보고**한다. 원자 신설은 디자인 판단이라 커서 레인이 아니다.

### 🚫 2-2. 공용 원자를 갈아치우지 않는다
이미 여러 화면이 쓰는 공용 컴포넌트를 **자체 컴포넌트로 교체 금지.**
개선이 필요해 보여도 보고만 한다. 하나를 바꾸면 그 원자를 쓰는 모든 화면의 톤이 깨진다.

> **실제 사고:** 4개 업무 페이지 중 3개의 목록이 공용 `ListRow` → 자체 `FeedListRow`(좌측 56~72px 아이콘 + 3줄 고정)로 교체됨. 정책관리만 남아 **4개 화면 톤이 갈라지고**, 행이 커져 한 화면에 보이는 건수가 줄어 스캔성이 떨어짐. 재고 목록은 차 사진 자리에 자동차 아이콘이 들어감.

### 🚫 2-3. 페이지에 레이아웃을 손으로 짜지 않는다
업무 페이지는 **4패널 고정**이며 반드시 `WorkPage`에 `panes` 배열로 선언한다.

| 페이지 | ① | ② | ③ | ④ |
|---|---|---|---|---|
| 계약문의 (`app/chat`) | 목록 | 채팅 | 상세(진행되면 첨부로 전환) | 진행 |
| 계약정산 (`app/contract`) | 목록 | 진행 | 첨부 | 정산상태 |
| 재고관리 (`app/inventory`) | 목록 | 기본정보 | 운영정보 | 업로드 |
| 정책관리 (`app/policy`) | 목록 | 기본·심사 | 계약조건 | 보험 |

패널 배치·모바일 전환·목록 도구(검색/정렬/필터)는 **셸이 담당**한다. 페이지는 내용만 넣는다.

### 🚫 2-4. v3(운영 중) 데이터를 건드리지 않는다
- v3는 **읽기 전용**. 쓰기는 전부 `v4/` 오버레이로만 간다(`lib/firebase/rtdb-adapter.ts`).
- v3 노드에 직접 write하는 코드를 **절대 추가하지 않는다.**
- `database.rules.json` 수정·배포 금지 (보안 규칙은 사람이 판단).

### 🚫 2-5. 엔진을 우회하지 않는다
상태·금액·계약 단계는 **엔진을 통해서만** 바꾼다.

| 대상 | 반드시 통해야 하는 곳 |
|---|---|
| 계약 단계 체크·완료·취소 | `applyStepCheck` / `cancelContract` (`lib/domain/settlement-engine.ts`) |
| 차량 상태(출고/계약중) | 엔진이 단일 writer — 페이지에서 직접 바꾸지 말 것 |
| 정산 금액 | 정산 엔진 |
| 저장·수정·삭제 | `getStore()` (감사·소프트삭제가 여기 붙어 있음) |

직접 `store.update`로 상태를 바꾸면 감사 기록·연동이 깨진다.

## 3. 반드시 지킬 것

### 3-1. 모바일 규격 (하드코딩 금지)
- 입력 폰트는 **모바일 16px** (미만이면 iOS에서 화면이 확대됨 = 버그)
- 터치 타깃 **40px 이상**
- 크기는 직접 쓰지 말고 `ctrlH()` / `ctrlInputFs()` / `useIsMobile()` 사용
- 원자를 쓰면 이 규격은 자동으로 지켜진다 → **또 하나의 "원자를 써야 하는 이유"**

### 3-2. ERP 디자인 도그마
- **목록이 제품의 본체** — 행·열·**스캔성** 우선. 한 화면에 많이 보여야 한다.
- **고밀도** 기업 백오피스 톤. 큰 카드·큰 아이콘·여백 과다 = 잘못된 방향
- **B2C 피드 패턴 금지** (당근류 큰 썸네일 피드). 이건 ERP 화면이다.
- 요약·KPI 큰 칸을 먼저 만들지 않는다. 요약은 정렬·필터로 창발시킨다.

### 3-3. 데이터 필드 계약 (전환 정합성)
계약·방·정산을 새로 쓸 때 **당사자 3필드를 항상 함께** 기록한다.
`agent_uid` · `agent_channel_code` · `provider_company_code`
→ v3는 `agent_uid`로 접근 권한을 판정한다. 비우면 전환 후 그 레코드는 아무도 못 본다. **빈 문자열로 채우지 말 것.**

### 3-4. 확정된 것은 건드리지 않는다
잘 돌아가던 기능·정책·UI는 **명시 요청 없이 변경 금지.** 버그를 발견해도 고치지 말고 **보고 먼저.**

## 4. 작업 전 체크리스트

1. 이 작업이 **배열인가, 디자인/설계인가?** → 디자인·설계면 멈추고 보고
2. 쓰려는 UI에 **공용 원자가 있는가?** → 없으면 멈추고 보고
3. 건드릴 파일을 **다른 작업자(Claude)가 방금 수정했는가?** → **편집 전 반드시 다시 읽는다**
4. v3 노드에 쓰는 코드가 포함되는가? → 있으면 중단
5. 상태·금액 변경이면 엔진을 통하는가?

## 5. 작업 후 체크리스트

1. `npx tsc --noEmit` 통과
2. 새로 만든 `<input>/<textarea>/<button>`이 없는지 확인 (숨김 파일선택 제외)
3. 공용 원자를 교체하거나 삭제하지 않았는지 확인
4. 모바일에서 입력 폰트 16px·터치 40px 유지되는지 확인
5. **바꾼 것과 그 이유를 요약 보고** (임의 개선이 섞이지 않았는지 스스로 점검)

## 6. 막혔을 때

> **새로 만들지 말고 멈추고 보고한다.**

- 원자가 없다 → 보고 (신설은 디자인 레인)
- 규격이 모호하다 → 보고 (추측 금지)
- 더 좋은 방법이 보인다 → **하지 말고** 제안만 보고
- 기존 코드가 이상하다 → 고치지 말고 보고

**"일단 되게 만들었습니다"가 가장 큰 손해다.** 되긴 하지만 톤이 깨지고, 나중에 전부 다시 걷어내야 한다.

## 7. 요약 (이것만 기억해도 됨)

1. 배열만 한다. 디자인·설계는 안 한다.
2. UI는 손으로 그리지 않는다 — 원자를 쓴다. 없으면 멈춘다.
3. 공용 원자를 갈아치우지 않는다.
4. 업무 페이지는 `WorkPage` 4패널로만.
5. v3에 쓰지 않는다. 엔진을 우회하지 않는다.
6. 모바일 입력 16px·터치 40px.
7. 막히면 새로 만들지 말고 보고한다.
